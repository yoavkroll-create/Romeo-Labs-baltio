# Prototype Validation — Definition of Done

## Criteria

| # | Criterion | Description |
|---|-----------|-------------|
| 1 | **Validation inputs gathered** | PM has provided feedback on the prototype — either from a built prototype (screenshots, walkthrough, user testing) or a spec-level review. |
| 2 | **All 6 dimensions scored** | Problem-Solution Fit, User Flow Clarity, Product Experience, Feature Completeness, Data & Integration, and Demo Readiness are each scored 1–5 with specific justification. |
| 3 | **Scores are evidence-based** | Every score cites specific observations, user feedback, or spec analysis — not gut feeling. |
| 4 | **Feature completeness assessed** | Every MVP feature is checked against the prototype with a Working/Partial/No status. |
| 5 | **Critical issues identified** | All blocking issues are listed with specific descriptions and suggested fixes. |
| 6 | **Scope adjustments documented** | Any feature priority changes (add/remove/reprioritize) are listed with evidence-based reasons. |
| 7 | **Next action determined** | Clear recommendation: proceed to Final PRD (score ≥ 4.0), iterate (3.0–3.9), or re-scope (< 3.0). |
| 8 | **Validation report saved** | Report is saved to `prototype/validation/validation-{variant}-round-{N}.md` with correct metadata. |

## Evaluation

- All 8 criteria must pass.
- The validation is honest and critical — generous validation that misses real issues is a failure.
- If the prototype hasn't been built, a spec-level validation is acceptable but must be clearly labeled as such.
- MVP validation is required for stage progression. Future validation is optional.
